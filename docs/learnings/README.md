# Ingestion & retrieval learnings — index

*Living doc set. One page per theme. Every rule earns its place by naming the incident, investigation, or PR that created it. Adding a new learning? Cite the commit or the §-reference in `docs/done/next/ingestionfix_v{1..6}.md`, `docs/done/next/ingestion_tunningv{1,2}.md`, the active forward plan `docs/aa_next/next_vN.md`, or the relevant architect study.*

**Scope.** The **headless ingest pipeline** (`src/lia_graph/ingest.py`, `ingest_classifier_pool.py`, `ingest_subtopic_pass.py`, `ingestion/`), the **served retrieval path** (`src/lia_graph/pipeline_d/`), and the **process discipline** we use to run both — investigations, monitoring, cloud writes. Not scoped: UI-facing ingestion, which has its own doc at `docs/done/next/UI_Ingestion_learnings.md`.

**Why this exists.** Between 2026-04-17 and 2026-04-24 we shipped five ingestion-fix waves (`ingestionfix_v1`–`v5`) followed by a v6 investigation+execution cycle (`ingestion_tunningv1` → `ingestion_tunningv2`). The next_v3 close (2026-04-25) added taxonomy v2 + classifier rewrite + K2 path-veto. The next_v4 same-day ship (2026-04-25) added the conversational-memory staircase Levels 1+2 and the `comparative_regime_chain` query mode. Each wave fixed a distinct class of failure. The NEXT PR that touches ingestion or retrieval is one careless code review away from bringing any of those failures back — because the fixes live in code, not in a doc the next contributor will read. These pages are that doc.

**Reading order for a cold engineer.**

1. [`process/investigation-discipline.md`](process/investigation-discipline.md) — why we run a week of read-only investigation before writing code. The v6 cycle would not have landed without this discipline.
2. [`ingestion/corpus-completeness.md`](ingestion/corpus-completeness.md) — the single highest-leverage finding across the entire program: *your retrieval is only as good as the corpus you ingested*.
3. [`ingestion/parallelism-and-rate-limits.md`](ingestion/parallelism-and-rate-limits.md) — how we got from 40 RPM sequential to 300 RPM parallel without introducing non-determinism. Includes the TPM-ceiling gotcha we hit on the cloud sink.
4. [`retrieval/coherence-gate-and-contamination.md`](retrieval/coherence-gate-and-contamination.md) — the biofuel-in-labor contamination case and why an evidence-coherence gate catches it where a classifier-confidence gate cannot.
5. The rest in any order, as the occasion arises.

- [`process/deep-trace-before-hypothesis-debate.md`](process/deep-trace-before-hypothesis-debate.md) — when the entire causal path from input to symptom is in code you can read, a code-anchored end-to-end trace often collapses competing hypotheses faster than empirical A/B testing. The next_v4 §3 trace ruled out H1 and structurally confirmed Frontiers 1+2+3 in one pass.
- **NEW** [`process/skill-as-verification-engine.md`](process/skill-as-verification-engine.md) — when an external SME delivers a complete protocol design, treat it as gate-1+gate-2 closure for the relevant fix. The vigencia-checker skill (1428 LOC, 8 files) collapsed ~3 weeks of design work for Fix 1A. Cost-side surprise: skill mandates double-primary-source per article → ~$83K of new scraper + harness + eval work; mostly absorbed by reserve. Burden-of-proof inversion as engineering discipline (`VigenciaResult.veredicto: Vigencia | None` — None means refused, not a confidence threshold). Generalizes to other domains (labor, cambiario, municipal).
- **NEW** [`process/activity-as-surgical-precursor.md`](process/activity-as-surgical-precursor.md) — Activities (< 1-day surgical ships) measure the structural hypothesis before committing to a multi-week Fix. This round: Activity 1 → Fix 1C; Activity 1.5 → Fix 1B-β + Fix 6 (revealed corpus hallucination); Activity 1.6 → Fix 5 (3 pre-validated test cases). Two patterns: "Activity reveals a bigger problem" (upgrade) and "Activity confirms the boring case" (greenlight). Six hours of Activities saved $80K of speculative Fix work.
- **NEW** [`process/re-verify-cron-criticality.md`](process/re-verify-cron-criticality.md) — for any product whose correctness depends on a moving target (Colombian tax law, regulatory environments, etc.), automated re-verification is launch-critical infrastructure, not Fix 6 polish. Activity 1.5 found T-I corpus document was correct in March 2026 but stale by April 2026 (Sentencia C-079/2026). Cron cost ~$8K vs launch risk of brand-damage tweet. Move from week 13 → week 4.
- **NEW** [`process/persistence-granularity-discrimination.md`](process/persistence-granularity-discrimination.md) — Activity 1.5b lesson on picking the right persistence target per fact: document-level / chunk-level / Falkor edge-level. Vigencia of an article is chunk-level (`chunk_id LIKE '%::<article>'`). Relationships between norms (X derogates Y, sentencia strikes down decreto) are edge-level — Falkor schema natively supports this; Supabase columns flatten it. Includes the 7-state → persistence-grain contract Fix 1B-γ implements at corpus scale.
- **NEW** [`process/h0-fakes-vs-live-db-gap-2026-04-27.md`](process/h0-fakes-vs-live-db-gap-2026-04-27.md) — H0 fake clients verify Python shape, not schema contract. First live e2e of fixplan_v3 1B-γ migrations caught 4 real bugs that 222 H0 unit tests missed. Every sub-fix that ships a new SQL function / UPSERT / CHECK / `text[]` parameter must ship a live-DB integration smoke gated on `LIA_INTEGRATION=1` alongside the H0 fake tests.
- **NEW** [`process/batched-test-runner-discipline-2026-04-27.md`](process/batched-test-runner-discipline-2026-04-27.md) — `scripts/run_tests_batched.py` rebuild + protocol.
