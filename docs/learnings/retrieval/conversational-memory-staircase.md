# Conversational memory — climb a staircase, don't jump to LLM summary

> **Captured 2026-04-25** from `docs/aa_next/next_v4.md §3 + §4`. Code lives in `frontend/src/features/chat/requestController.ts:230-243`, `src/lia_graph/pipeline_c/conversation_state.py:102-220`, `src/lia_graph/topic_router.py:639-919`, `src/lia_graph/ui_chat_payload.py:299-460`.

## The symptom

Two-turn dialogue against the live UI, with `LIA_TEMA_FIRST_RETRIEVAL=on` + `LIA_EVIDENCE_COHERENCE_GATE=enforce`:

- **T1.** Long consulta about pérdidas fiscales → routed `perdidas_fiscales_art147`, art. 147 ET answered cleanly.
- **T2 (follow-up, ~12 tokens).** "existe algun limite de monto de dinero a descontar por año?" → classifier picked `descuentos_tributarios_renta`; retriever still pulled art. 147 ET; coherence gate fired with `primary_off_topic` and the user got the abstention text.

Retriever was right. Classifier was wrong. The gate did what it was designed to do (Q1-class contamination guard) but fired on a follow-up the system had enough state to answer.

## The mechanism — three serial frontiers

A code-anchored deep trace from browser keystroke to refusal showed the prior turn's `effective_topic` was being dropped at **three independent boundaries, in series**. Any single-frontier patch is insufficient at one frontier and unnecessary at another:

| # | Frontier | Site | What's broken |
|---|---|---|---|
| 1 | FE `/api/chat` body omits `topic` | `requestController.ts` | Backend `ui_chat_payload.py` reads `payload.get("topic")` → `None` |
| 2 | `ConversationState` dataclass has no `prior_topic` slot | `conversation_state.py` | `build_conversation_state` walks turns but never reads `turn_metadata["effective_topic"]`; the persistence layer DOES save it (`ui_chat_persistence.py:57,78,80`) — storage-only, not state-flow |
| 3 | `resolve_chat_topic` signature has no `conversation_state` parameter | `topic_router.py` | `_build_classifier_prompt` cannot embed prior context even when state is available |

The persistence half-built chain was the most surprising finding: the data was being saved to the database every turn, just never read back. **State-flow is not state-storage.**

## Why H1 (lower the confidence threshold) was the wrong fix

`topic_safety.detect_topic_misalignment` and `_coherence_gate.detect_evidence_coherence` Case A evaluate the same condition over the same data — the second receives the first's verdict and forwards it when primary_articles is populated. Their visible difference is only the refusal *text*: topic_safety fires when confidence < 0.50; coherence_gate fires when confidence ≥ 0.50.

**Even if you lower `MISALIGNMENT_PROMOTE_TO_ABSTENTION_BELOW`, the answer is still an abstention — just with a different sentence.** The user is not blocked by which text shows; they are blocked by the system refusing at all. The fix has to be upstream of both gates: repair state-flow so the gates see aligned data.

## The staircase — measured architectural escalation

The instinct to "add an LLM rolling summary so the next question always reads from a continuously-updated story of the conversation" is on-trend (ChatGPT / Claude.ai / Cursor do variants). For an accountant-facing product where citation traceability is the trust contract, a free-text summary introduces **drift, latency, and debug-cost** that don't compensate the marginal benefit when 80% of the value can be unlocked with structured-state extensions of an existing primitive.

**Climb the staircase. Only escalate when the previous level demonstrably can't reach.**

| Level | What | Cost | What it unlocks | What it does NOT unlock |
|---|---|---|---|---|
| **1 — FE topic propagation** | Forward `payload.topic = lastTurnEffectiveTopic` | ~14 LOC FE | Short follow-ups stop being silently refused when prior turn anchored a clear topic | Multi-turn drift; subtopic shifts; parallel topic tracks |
| **2 — Structured `ConversationState` extension** | Add `prior_topic` / `prior_subtopic` / `topic_trajectory` / `prior_secondary_topics` to dataclass; `build_conversation_state` reads from `turn_metadata`; `resolve_chat_topic` accepts `conversation_state` and uses `prior_topic` as a soft tiebreaker | ~80 LOC + tests + state schema migration | Server-side reconstruction immune to FE drift; future planner/synthesizer features can read trajectory; future fields (entities, carry_forward_facts) trivial to add | Conversations where the user is reasoning across multiple norms simultaneously and a free-text gist would help |
| **3 — Free-text LLM rolling summary** | Per-session running narrative, generated by an LLM after each assistant turn, fed into classifier+planner+synthesizer | ~2-3 weeks (incl. anti-drift harness) + ongoing per-session token cost | Multi-turn semantic continuity beyond what structured state captures | (deferred — no current need) |

Levels 1+2 shipped same-day 2026-04-25. Level 3 is **deferred** to v5+ with binding reopen conditions.

## Binding reopen conditions for Level 3

Level 3 reopens only when **all three** are true:

1. Level 2 ships and the multi-turn harness still shows refusal rate > 15% on 5+ turn dialogues.
2. SME spot-review identifies ≥ 3 distinct failure cases where structured state demonstrably could not have captured what was needed (e.g., user reasoning across multiple norms simultaneously).
3. Operator explicitly opens v(N) §X with a six-gate plan that includes an **anti-drift measurement harness BEFORE any code is written**.

Level 3 has known failure modes that need their own measurement infrastructure: drift / telephone-game effect (a turn-N summary that mis-compresses propagates to N+1, N+2…), latency cost (+1-3s per turn), debug surface (free-text prose vs structured state — ~10× debug-time multiplier), citation traceability risk (paraphrasing legal positions can shift the answer's framing).

## The implementation in one paragraph

**Frontier 1 (FE).** `submitChatTurn` reads the most recent `state.questionEntries[i].effectiveTopic` and forwards it as `payload.topic` whenever a prior assistant turn anchored a topic. **Frontier 2 (state).** `ConversationState` gained four fields; `build_conversation_state` lifts `effective_topic` / `secondary_topics` / `effective_subtopic` from each assistant turn's `turn_metadata`. **Frontier 3 (classifier).** `_build_classifier_prompt`, `_classify_topic_with_llm`, and `resolve_chat_topic` accept optional `conversation_state`. The prompt embeds `prior_topic` as a soft hint mirroring the existing `requested_topic` retention rule. A strict tiebreaker fires only when (a) lexical scoring is empty AND (b) the LLM verdict diverges from `prior_topic` AND (c) `prior_topic` is in `_SUPPORTED_TOPICS` — returns the prior with confidence boosted +0.15 capped at 0.85, mode `prior_state_tiebreaker`. **Wiring.** `ui_chat_payload.py` does a best-effort early `load_session` peek to extract `conversation_state` before `resolve_chat_topic`; the loaded session is cached on `request_context` so `_ensure_conversation_session_loaded` later doesn't double-pay the IO.

## Anti-patterns

- **"Just lower the abstention threshold."** The refusal text would still fire — different sentence, same blocked answer. Always trace which gate fires before tuning thresholds.
- **"Add an LLM rolling summary."** Plausible-sounding; fails the cost/risk analysis at internal-beta scale. Climb the staircase first.
- **"Patch one frontier and ship."** When state-flow has multiple serial breaks, single-frontier fixes are fragile. Ship A as primary + B/C as defense-in-depth on the same task.
- **"The data is in the database, must be available."** Storage ≠ state-flow. Verify the read path actually reaches the consumer.
